1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional
requirements for NOVA (Navigational Object and Voice Assistant), a low-cost assistive vision mobile
application designed for blind and visually impaired (BVI) users in Cameroon and broader
Sub-Saharan Africa. NOVA targets Android devices running API level 26 (Android 8.0) and above,
and is implemented using the Flutter framework.
This document is intended for use by the development team, academic supervisors, and future
maintainers of the system. It serves as the authoritative reference for all design, implementation,
testing, and evaluation decisions made throughout the project lifecycle.

1.2 Scope
NOVA is a smartphone-based assistive vision system that uses the Android device camera, on-device
machine learning inference, cloud-based inference for heavy tasks, and text-to-speech output to help
BVI users navigate their environment, read text, identify currency, recognise known faces, and obtain
scene descriptions.
The system comprises four integrated components:
•​

A Flutter mobile application (Android) that captures camera frames, runs on-device ML
inference, manages TTS output, and communicates with the backend.

•​

A FastAPI backend server that handles computationally expensive inference tasks (scene
description, face recognition against server-side galleries) and serves updated ML models.

•​

A PostgreSQL relational database storing user profiles, enrolled faces, usage logs, and
feedback.

•​

An ML pipeline (offline) producing quantized TensorFlow Lite models for obstacle detection
and currency detection, deployed to the mobile app.

The following are explicitly out of scope: iOS support, clinical diagnostic features, BLE beacon
infrastructure, production-scale deployment, and longitudinal field studies beyond the project
evaluation period.

2. Overall Description
2.1 Product Perspective
NOVA is a new, standalone mobile application positioned in the ecosystem of smartphone-based
assistive technology. Unlike existing solutions such as Microsoft Seeing AI (iOS-first), Envision AI
(cloud-dependent, subscription model), and OrCam MyEye (USD 4,500 dedicated hardware), NOVA
is designed from the ground up for:
•​

Low-end Android devices (quad-core CPU, 2 GB RAM, no dedicated NPU) dominate the
Cameroonian market.

•​

Intermittent mobile data, all safety-critical features operate fully offline.

•​

The CFA franc currency zone and bilingual French/English Cameroonian context.

•​

A zero-cost-to-user model, the application is free with no subscription.

NOVA follows an IoT architecture: the Android smartphone acts as the edge endpoint for real-time
inference; the FastAPI backend acts as the cloud service for heavy computation; and an operator
dashboard (separate from this SRS) monitors system health and usage.

2.3 User Classes and Characteristics
2.3.1 Primary User Blind or Visually Impaired Individual
The primary user is an adult (18–65) who is bilaterally blind or has severe visual impairment, owns or
has access to an Android smartphone (API 26+), and communicates in French and/or English. This
user interacts with NOVA exclusively through voice commands and audio output. They cannot rely on
any visual interface element. Their primary needs are: safe navigation (obstacle avoidance), reading
printed text in their environment, identifying banknotes, recognising familiar people, and
understanding scenes they cannot perceive.

2.3.2 Secondary User Caregiver or Support Person

A caregiver or family member may assist with initial app setup, face enrolment, and emergency
contact configuration. This user may have partial or full vision and can interact with visual interface

2.3.3 Operator / Developer
A developer or project team member accesses the backend operator dashboard to monitor usage logs,
model performance, and aggregate feedback. This user interacts with the system entirely through the

2.5 Design and Implementation Constraints
•​

All safety-critical features (obstacle detection, OCR, and currency detection) MUST function
with zero network connectivity.

•​

On-device quantized model size per feature module MUST NOT exceed 20 MB.

•​

On-device inference latency for obstacle detection MUST remain below 100 ms per frame.

•​

The application MUST NOT transmit raw camera frames to the server at any time unless the
user explicitly triggers a cloud feature (scene description, cloud face matching).

•​

All user-facing interactions MUST be operable via TalkBack screen reader and voice
commands; no critical function may require visual feedback.

•​

The application MUST support Android TalkBack out-of-the-box (ContentDescription on all
interactive elements).

•​

Battery consumption during continuous active use MUST remain below 25% per hour on the
reference low-end device.

•​

The app MUST be distributable as a standard APK without requiring Google Play Store
access.

2.6 Assumptions and Dependencies
•​

Users have a functional Android device with a working rear camera and speaker.

•​

Enrolled face images are provided by the user or caregiver during setup; NOVA does not
perform unsolicited face surveillance.

3. Functional Requirements
3.1 Module MOD-01: Obstacle Detection
3.1.1 Overview
MOD-01 provides continuous, real-time detection of obstacles in the user's forward field of view
using the device's rear camera. The module runs a quantized object detection model on-device and
issues directional audio alerts through the device speaker (or connected audio output) when obstacles
are detected within configurable distance thresholds. MOD-01 is the highest-priority safety feature
and must function fully offline.

3.3 Module MOD-03: Scene Description
3.3.1 Overview
MOD-03 generates a natural-language description of the scene visible to the camera. This feature is
cloud-dependent when triggered; a compressed still frame is sent to the FastAPI backend, which
passes it to a vision-language model (VLM) and returns a text description that NOVA reads aloud.
MOD-03 is explicitly not a safety feature and is documented as cloud-dependent to the user.

3.5 Module MOD-05: Face Recognition
3.5.1 Overview
MOD-05 allows the user to enrol known contacts (family members, colleagues, caregivers) by
providing photos during setup. When the user triggers face recognition, the front or rear camera scans
the scene, detected faces are matched against the enrolled gallery, and recognised individuals are
announced by name via TTS. Detection occurs on-device; matching against the enrolled gallery may
be performed on-device (if the gallery is small, <= 20 faces) or via a backend API call (for larger
galleries).

5. External Interface Requirements
5.1 User Interfaces
NOVA's mobile UI is designed around the principle of audio-first interaction. Visual elements are
present for sighted caregivers and setup scenarios, but are never the sole interface for any function.
•​

Main Menu: Five large, high-contrast buttons (one per module) with icons and spoken labels.
TalkBack-navigable. Voice shortcut overlay always active.

•​

Settings Screen: Voice-navigable list of configurable parameters (TTS speed, thresholds,
language, emergency contact). All changes confirmed via TTS.

•​

Enrolment Screen (MOD-05): Camera viewfinder with audio guidance for centering the face;
the sighted caregiver can verify the frame before capture.

•​

Status Overlay: A minimal always-visible strip showing active module and connectivity
status, readable by TalkBack.

for routing to Bluetooth if connected.

•​

Vibration Motor: Accessed via Flutter vibration plugin for haptic alerts.

•​

GPS: Accessed via Flutter geolocator plugin for emergency location sharing (MOD

6.2 Data Retention Policy
•​

Raw camera frames: Never persisted. Processed in memory; discarded after inference.

•​

Scene description images: Transmitted to backend for processing; not stored after response is
generated.

•​

Face enrolment images: Used to generate embeddings; not stored after embedding extraction
on the backend.

•​

Usage event logs: Retained on the backend for 12 months for analytics; anonymised after 12
months.

•​

Face embedding vectors: Retained until the user explicitly deletes the enrolled contact.

•​

JWT refresh tokens: Expire after 30 days; invalidated on explicit logout.
